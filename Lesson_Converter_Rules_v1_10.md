## 0. Philosophy

- **Source-first.** Pour the source's lesson content **faithfully**. Do not generate, summarise, or "improve" it.
- **Strict chrome, flexible body.** The template's **chrome is binding**: title bar / banner wording-format, icon, colour palette, and layout furniture follow the house-style template **exactly**, every lesson — this is what makes the output look consistent. The template's **body content is a scaffold, not a cage**: use it only where the source content genuinely maps to it. Where it doesn't fit, **adapt the body to the actual content** rather than shoe-horning the content into the template's slots (don't truncate, pad, reword, or drop source content to make it fit). When the body is adapted beyond a straight pour, mark `⚠ ADAPTED: <what changed>` so the user can review.
- **Title bars are the strict exception.** Banner text follows the template's canonical format (e.g. `FRAYER MODEL: <topic>`, `Checks for understanding`, the I/We/You-do label band) even when the source phrased its heading differently — consistency of the title bar trumps source wording.
- **Live shapes stay live — never rasterize as a shortcut.** When the source's content is built from live shapes/text (annotated diagrams, text columns, dimension lines, callouts) with no embedded image, **rebuild it as live shapes/text** on the target template. Do **not** flatten a source slide to a picture to save effort — not even to preserve annotations. (Full rule: §2.13.)
- **Insert only genuinely missing structural slides.** Flag content gaps as `⚠ PLACEHOLDER`, never invent.
- **Re-skin = transplant + pour (+ adapt).** Take the correct template slide, transplant it over the source slide, pour the source's verbatim text/media into the matching slots, and adapt the body where the fit isn't clean (per above).
- **Leave-as-is only as a last resort.** A slide is left untouched only when no template type matches *and* the slide already reads acceptably. The type-driven mapping in §1 should catch almost everything; genuinely unclear cases go to the §1.1 check-in.
- **Always be learning (capture every steer).** Whenever the user provides *any* additional steer during a conversion — answering an ambiguous check-in, overriding a "clear" call, correcting a mid-build mapping, naming a new icon→template signal, or stating a handling preference — that steer is a candidate durable rule. **Two things must happen, every such conversation** (see §4 for the pipeline step): (1) **write it to memory** so it informs future conversions immediately, and (2) **capture it as a generalized bullet at the end of the conversation** — phrased as a detection-signal → target-template (or handling) rule, topic-and-slide-number agnostic — ready to fold into the next rules version. This applies to **every** conversion that yields new routing/handling information, not just major ones. If a new steer **contradicts an existing rule**, or if **memory and this document disagree**, do not silently overwrite either — surface it as an `⚠ AMBIGUITY` and let the user resolve it (never auto-resolve in favour of memory or the doc). See §5 for the full loop and precedence policy.

**Gap-fill / cloze image exercises** (passages with `___` blanks delivered as baked-in images) have no single-line Q/A template. **Information / knowledge slide decision (the key clarifier).** Many source slides are explanatory diagrams or text. Classify them by *furniture*, not appearance:
1. **Hand-holding-jigsaw icon in the title banner?** → Collect-the-evidence → **HS14**.
2. **No title at all?** → it is an information slide (proceed to 3).
3. **Carries a clear question or task instruction?** → **HS13** (info slide *with* task). **Otherwise** → **HS7** (info slide, no question/task — the default knowledge slide).
4. Specific furniture overrides the above: a **partner task** ("Window/Door partner") with no TPS bars → HS12; a **chart/graph + Tip** *or* the **SKILL spanner+pencil icon** → HS15; a **Keywords-list prose** slide → HS16; **TPS header bars** (with or without a reveal) → HS9; a **mini-whiteboard icon** → HS20; a **hand-holding-jigsaw icon** → HS14.

**Icon-first routing (the reliable signal).** Source decks are inconsistent in layout but consistent in their **furniture icons**, so when an icon is present it routes the slide regardless of how heavy or light the body is:
- **mini-whiteboard icon** → HS20 (not HS13/HS7 — booklet work, which lacks the icon, is HS13).
- **SKILL spanner+pencil icon** → HS15 (adapt the body down; don't reject HS15 for being chart-heavy).
- **hand-holding-jigsaw icon** → HS14 (collect-the-evidence).
- **KEYWORD magnifying-glass icon** → HS19 (keyword-definition prose, §1 row 17c).
- **TPS header bars** → HS9 (vs HS8 only when the *full* TPS boxes with timings are present); the bars, **not** a reveal popup, are the HS9 signal.
- **I/We/You-do label band** → HS24/25/26.
Icons override body-fit. Where the icon's template scaffold is heavier than the source body, keep the chrome and adapt the body down (flag `⚠ ADAPTED`).

**Two-column slides — three-way split (common ambiguity).** A slide with two parallel columns can be any of three things; disambiguate by whether there's a task and where it lives:
- **Spec table** (left = key idea, right = spec content) → **HS6** (§ row 5), even with an arrow between the blocks.
- **Task embedded in the body text** (no separate task box) — e.g. Primary\|Secondary effects, Immediate\|Long-term responses, or a muddled-sort activity → **HS13** (§ row 12).
- **Sorted answer / parallel info, no task** → **HS7** (§ row 6). A separate spec-style two-block that isn't a spec table and isn't task-bearing → **HS4** (§ row 5b).

### 1.1 — Match confidence & the user check-in *(mandatory)*

Source decks are inconsistent and will not all map cleanly. For each source slide, assign a **match confidence**:

- **Clear** — one template is an obvious fit and the body content maps to its slots without distortion.
- **Ambiguous** — *any* of: no template fits well; two or more fit plausibly; the chrome matches but the body content only partially maps; or the slide is an odd one-off the taxonomy doesn't anticipate.

Then:

1. Produce the **full mapping table** (every source slide → proposed template + a one-line note), so the user sees the whole plan. Clear matches are listed for transparency and proceed without interrogation — **do not** make the user confirm obvious matches one by one.
2. **Surface only the ambiguous ones for a check-in.** For each, present: a one-line summary of the source slide, **1–3 candidate templates with the rationale for each**, and let the user **confirm one, choose another, request a generic info template (HS7/HS13), or mark it `⚠ PLACEHOLDER`**. Offer this as tappable options where the interface allows.
3. **Wait for the user's decision on every ambiguous slide before building.** Never silently guess an ambiguous match. The user may also override a "clear" call — the table is theirs to redirect.

This check-in is cheap (a short table + a few choices) and prevents the expensive failure of building a slide onto the wrong template.

## 2. Content & styling rules

**2.1 — Question → Answers content-parity / stale-answer / empty-shell check.** When a slide is immediately followed by its "Answers" slide (e.g. Do it Now → Do it Now Answers), the **base prompts must match** between the two. After conversion, diff the question text of the pair; if the prompts differ, raise `⚠ PARITY: Q/A base content differs` rather than silently shipping. Recurring special cases:
- **Stale answers from the previous lesson.** If the Do-it-Now Answers slide (HS3) carries leftover Q+A from the *previous* lesson, the user's preference is to **copy the matching Do-it-Now (HS2) prompts onto it and supply genuine answers for THIS lesson**, rather than ship the stale content.
- **Empty shell.** If the HS2 Do-it-Now is an **empty shell** (EQ filled but THIS UNIT / PREVIOUS UNIT / SKILLS FOCUS carry no prompts), there is nothing to copy or answer. **Surface this and check in** — do not invent questions. Default when the user opts for a clean shell: pour the EQ and set every prompt and every answer to `…` placeholders on both HS2 and HS3.
- **Leftover / incomplete standalone "Answers" slide.** A bare "Answers" slide that only restates a few leftover or incomplete answers from booklet/MCQ work (e.g. "1. A, 2. B" with the rest missing) maps to **HS7 framed as an answer sheet** — **not** HS21/HS3 (which are mini-whiteboard reveals). **Pour exactly what the source gives and flag the incompleteness** (`⚠ PLACEHOLDER: source answer sheet incomplete`); never invent the missing answers to "complete" it.

**2.2 — Video slides always keep a Keywords box.** Every video slide retains the template's Keywords box. If the source supplies keywords, pour them in. **If it does not, populate the box with a bold `Keywords:` header and `…` on the line below** — never leave the box empty or delete it. This keeps the font/size correct for the teacher to type into, and lets them delete the whole box in one action if unwanted.

**2.3 — Video link/URL always carried.** Preserve the source's video link. Render it under a bold `Video link:` label as a live hyperlink. If the source link is reference text (e.g. "Use this link: … 7 mins"), keep the wording but still wire it as a hyperlink. Never drop the URL. (Carry the embed/watch URL from the source slide's `.rels`, and update both the visible run text **and** the hyperlink relationship target.)

**2.4 — Image sizing, placement & alignment — maximise, don't just fit.** Keep content images **as large as the slide allows** — this is a default, not an exception. The template slot gives the anchor/position, but a *content* image (the main map / photo / diagram — **not** the small banner icon) should be **scaled up to fill the available content area** (below the banner, inside the slide margins), preserving aspect ratio and centred, rather than left at the slot's default size. Whichever of width or height hits the content box first sets the scale.
- **Single content image:** fill the content area — roughly full slide width × the height between the banner and the bottom margin (on a 13.33×7.5″ slide that's ≈ up to 10–11″ × 5.5–6″, aspect-dependent).
- **Multiple images:** partition the content area (side-by-side or stacked) and size each to the largest that fits its share. Don't leave images small with empty space around them; rearrange to use the space.
- **Image beside a task/text box (alignment):** a large content image next to a task text box should be **TOP-aligned** with that text box, and sized as large as possible **without forcing the body/task text too small**. Balance the two: the image should dominate the content area, but the task text must remain comfortably readable.
- **Constraints:** preserve aspect ratio (never stretch), never overflow the slide edge, and don't scale a labelled diagram so far down that its labels become unreadable.
- If the source has more images than slots, enlarge/rearrange to use the space rather than dropping them; flag `⚠ EXTRA IMAGE` only if they genuinely can't fit.
- **Watch for template rotation:** clear any `rot` transform (`rot="0"`) before pouring a normally-oriented source image, or it displays sideways. Resize via `spPr/xfrm/ext` (§3.7).
- **Verify the image visually before transplanting — filenames are no guide.** A source image's filename and its position in `media/` order do **not** indicate its content: a `media/imageNN` may be a screenshot of a PowerPoint window, an inset, a small icon, or the wrong diagram. Before swapping a content image, **eyeball each candidate** (render or thumbnail it) and pick the one that matches the slide's geography/topic — never trust filename order. If the **only** available source image for a slide is itself a screenshot or low-quality capture, **transplant it faithfully but flag it** (`⚠ SOURCE IMAGE: screenshot/low-quality — recommend replacement`) rather than substituting cleaner material of your own (source-first, §0).

(Reference: NH3 slides 6–10 and 17–20 — the converter under-sized these; the target is fill-the-space, e.g. a content image grown from 5.1×3.6″ to 10.6×5.9″.)

**2.5 — Fit-to-box / font sizing.** Body text boxes use the template's default size **but must fit**. Rule: start at template size; if the verbatim content overflows the box, step the font down (in 2pt steps) until it fits, to a floor of 16pt; below that, raise `⚠ OVERFLOW`. Prefer enabling shape **autofit** (`normAutofit`) so PowerPoint also reflows on edit — **but see §2.11: autofit alone does not shrink poured text; a `fontScale` must be written.** Where content is short, nudge **up** toward the template max so the box isn't sparse.

**2.6 — Booklet-reference slides (HS22) carry no green/Knowledge banner.** These are navigation slides; suppress the knowledge/coloured banner entirely. Gap-fill cloze exercises routed to HS22 follow the same rule.

**2.7 — Answers in bold.** On any answer variant (HS21 answer, HS22 answer / cloze-answer), the revealed answer text is **bold** to distinguish it from the question.

**2.8 — Frayer model: plain (HS18) vs with side furniture (HS19).** A plain 4-quadrant Frayer → **HS18**. A Frayer that also carries a side prompt and/or image (e.g. "What is a plate margin or boundary?") → **HS19**, the dedicated v4 template with a side text box + image beside the quadrants — use it rather than improvising side furniture on HS18. In both, the quadrants are a real **table + oval shape** (not a flattened image): when resizing, resize the table and the oval **together**, and squeeze the oval toward a circle as the table narrows so the proportions stay right. (§3.7 applies — resize via `spPr/xfrm/ext`.) **Note:** HS19 is also the target for keyword-definition prose slides (§1 row 17c) — in that case strip the Frayer table + oval and pour definition prose into the body instead.

**2.9 — Reveal animations travel with their boxes.** *(See §3.2 — this is the structural counterpart.)* If a template's answer/keyword boxes are click-to-reveal, preserve the entrance animation when the box is kept, and add one (clone an existing click node) when a new reveal box is introduced. **When a reveal box is deliberately removed** (e.g. the TPS-no-reveal variant of HS9, §1 row 8), delete its animation too and collapse the empty timing shell (§3.2).

**2.10 — Variable objective rows on the EQ/LO slide (HS5).** HS5's "Today's lesson involves" is a **multi-row layout** (default 2 rows). The objective rows are ordinary text-box shapes (emoji box + text box) over rounded-rectangle backgrounds; the **Key Words list is a SmartArt** (`diagrams/data1.xml` + `drawing1.xml`). To match a source with a different objective count, **clone or remove a row triple** (background + emoji box + text box, with fresh `cNvPr` ids) and re-space all rows evenly. Pour the source's Key Words into the SmartArt by editing the keyword text in **both** `data1.xml` and `drawing1.xml` (the cached drawing); if the source/template keyword counts differ, add/remove SmartArt nodes (or blank surplus nodes) rather than overstuffing. Remember to copy all five diagram parts (`data1`, `layout1`, `quickStyle1`, `colors1`, `drawing1`), wire the `diagramData/Layout/QuickStyle/Colors/diagramDrawing` relationships, and add their `[Content_Types].xml` overrides — a transplant that copies only images will leave the SmartArt relationships dangling.

**2.11 — Enquiry Question bar (HS2/HS3) — 28pt base + autofit *with a written `fontScale`*.** The "Enquiry Question:" bar uses a **28pt** base with the box set to shrink-on-overflow. **Critical correction:** setting only `<a:normAutofit/>` does **NOT** shrink poured text. PowerPoint (and LibreOffice) render the text at full base size until the box is *manually edited in PowerPoint*, because the shrink factor (`fontScale`) is only computed and stored on that edit. A long question poured programmatically therefore **spills past the bar** even though autofit is "on". Fix: when pouring text that overflows the bar, **write a computed `fontScale` (and a small `lnSpcReduction`) onto the autofit element**, e.g. `<a:normAutofit fontScale="70000" lnSpcReduction="20000"/>` (values in 1000ths of a percent — `70000` = 70%). Size the scale so the wrapped text fits the bar height: estimate lines-at-base-size against the box's available height, step the scale down from 100% until it fits (floor ~25%), and write that value. PowerPoint honours the stored scale immediately and still re-fits on edit. **This applies to any `normAutofit` box where poured text overflows**, not just the EQ bar. On transplant, preserve the box's `normAutofit` and 28pt base; don't strip the autofit or rebase the font — *add the `fontScale`*.

**2.12 — Table formatting.** When a source table is poured onto a slide (data table, comparison grid, etc.), default to: **fill the available content space** (size the table to the content area, not a small default); body text **Calibri 18**; **header row bold**, its text **centred horizontally and vertically**, on a **shorter** header-row height with **sensible (taller) body-row heights**; **white table background/fill**. Exception: where a template defines its own table style (e.g. the **HS6** spec table with its Key idea | Specification content header), keep the template's styling and just pour the content. (Reference: NH3 slide 16.)

**2.13 — Live shapes over raster (never flatten a source slide to a picture).** When a source slide's content is built from live shapes/text — annotated cross-sections, labelled diagrams, dimension arrows, callouts, text columns — and has **no embedded image**, **rebuild it as live shapes/text** on the target template. **Never rasterize the source slide to an image as a shortcut**, even to preserve annotations. The faithful transplant is: extract the source slide's drawing elements (sp / cxnSp / grpSp, excluding the `nvGrpSpPr`/`grpSpPr` headers), renumber all `cNvPr` ids into a high unique range to avoid collisions with the template chrome, run the §3.8 whitespace sweep over the grafted runs, then wrap them in a single `<p:grpSp>` whose `chOff/chExt` match the source coordinate space and whose `off/ext` place + scale the group into the template's content band (below the banner, above the footer). Keep the template chrome (banner, icon, pill); drop the template's now-unused picture slots (and clean their rels/animation per §3.2–3.4).

**2.14 — Title-bar text: 32pt Calibri Bold, with a "..." placeholder when blank.** Title-bar / banner text is **always 32pt Calibri Bold**. If a source slide has **no obvious title text** (e.g. a no-title information slide), put **`...`** in the banner as a placeholder — **never leave the banner blank**. (The banner is binding chrome per §0; the placeholder keeps the bar visually consistent and gives the teacher an obvious edit point.)

**2.15 — TPS sharing icon stays.** TPS slides (**HS8** and **HS9**, including the no-reveal variant) **always keep the Think-Pair-Share sharing icon in the top-left of the title bar**. Do not drop it when transplanting or when repurposing the banner text.

**2.16 — Keyword term in bold.** On keyword/definition slides (HS16 Keywords-prose, HS19 keyword-definition prose, and any keyword list), the **keyword term itself is bold** (e.g. **Primary effects**, **Secondary effects**), with the definition that follows in normal weight.

**2.17 — Table-task twins & baked-MCQ-image handling (HS13).** Two recurring booklet-work shapes both route to HS13 (§1 row 12) and share handling:
- **Blank-table task + its "Answers" twin.** A source often pairs a *blank* characteristics/comparison table (the task) with a *filled-in* "Answers" version. **Both** become HS13. The blank version carries the table **image** + the booklet task instruction in the Task box. The answers version carries the **filled content**: prefer transplanting the source's live `<a:tbl>` as a graphicFrame (renumber its `cNvPr` id into a high range, reposition via `p:xfrm` into the content band, §3.7) and **remove the Task box + any GBR/template placeholder image**; if the answers exist only as an image, pour that image instead. Don't split the pair across a task template and a separate answers template.
- **Per-question MCQ image.** When each MCQ's question + options are **baked into a single image** (only the "In your booklets" title is live text), place each MCQ image as the **maximised content** on its own HS13 (§2.4), **remove the Task box** (the image is self-contained), and pour the standard title. One output slide per source MCQ. Do **not** try to re-key the options as live text or force a Q-only checks template (HS20/HS21).

## 3. Structural integrity & repair gates

These are **hard gates** — a converted deck must pass all of them before delivery. Several of these issues cause PowerPoint "repair" prompts that LibreOffice-based QA does **not** catch — notably the orphaned-animation/empty-timing-shell cases (§3.2) and the **whitespace-preservation** case (§3.8). All are *silent* under LibreOffice and `pack.py`'s normal render: the deck looks fine and even re-renders cleanly, yet PowerPoint still offers to repair on open.

**3.1 — Strip co-authoring `changesInfo` (and `revisionInfo`).** Decks edited in PowerPoint-web carry `ppt/changesInfos/changesInfo1.xml` (a change-log referencing shapes by slide-id/shape-id) and may carry `ppt/revisionInfo.xml`. Re-skinning invalidates them. On every convert, remove for **each**: the part itself, its `_rels`, the relationship in `presentation.xml.rels`, and the `[Content_Types].xml` override. (Watch for attribute-order variance when matching the `<Override>` lines — match on the `PartName`, not the whole literal element.)

**3.2 — Animation-orphan rule (the big one).** When a shape is **deleted** during conversion, also delete **every** animation reference to it:
- its `<p:par>` node(s) in `<p:timing>` (walk up from `<p:spTgt spid="…">` to the per-click `<p:par>` and remove it), **and**
- its `<p:bldP spid="…">` entry in `<p:bldLst>` (remove; if the `bldLst` becomes empty, remove the `bldLst` too).

An orphaned `spTgt`/`bldP` pointing at a missing shape is a guaranteed PowerPoint repair trigger. (This bites whenever a template carries reveal animations and you delete an unused box — e.g. an empty booklet text box on HS22, **or the removed reveal box on a TPS-no-reveal HS9, §1 row 8 / §2.9**.)

**Then collapse the empty timing shell.** Removing the last `<p:par>` from a slide's main sequence leaves a hollow `<p:timing>` — a `<p:seq nodeType="mainSeq">` whose `<p:childTnLst>` is now empty. That empty mainSeq is **itself** a repair trigger (and a silent one: orphan-spid checks pass because there are no targets left to be orphaned). Rule: after orphan removal, if a slide's `<p:timing>` contains no real animation (no `<p:par>` with an `<p:spTgt>` under the mainSeq), **delete the entire `<p:timing>` element**. A slide with no animations must have no `<p:timing>` at all. Add to the §3.5 gate: *no `<p:timing>` may contain an empty mainSeq `<p:childTnLst>`.*

**3.3 — Reveal preservation / cloning.** Keep a box → keep its `<p:par>` + `<p:bldP>` (they travel together). Add a reveal box → clone an existing click `<p:par>` + `<p:bldP>` and retarget the `spid`. Reveals are essentially free to preserve and cheap to add; the template carries the scaffold.

**3.4 — Diagram (SmartArt) & HD-photo parts travel whole.** When transplanting a template that uses SmartArt or `mc:AlternateContent` image layers, a copy-the-images-only transplant will leave parts dangling. Specifically: copy all SmartArt diagram parts and wire their relationships (§2.10); and if a picture has an `<a14:imgLayer r:embed>` HD-photo (`.wdp`) backing, copy the `.wdp`, add its relationship (type `…/2007/relationships/hdphoto`) and a `wdp` Default content-type — or remove the `imgLayer` and keep the plain blip.

**3.4b — Notes-slide references are 1:1.** A `notesSlide` part may be referenced by **exactly one** slide. When duplicating a template slide by any means other than `add_slide.py` (e.g. copying a pristine slide XML + its `.rels`), the copy will point at the **same** `notesSlideN.xml` as the original — a validation failure ("referenced by multiple slides"). Fix: strip the `notesSlide` relationship from the duplicate's `.rels` (the slide doesn't need notes), or give it its own notes part. `add_slide.py` handles this automatically; manual copies do not.

**3.5 — Pre-ship validation gate.** Assert all of:
- every `spid` in `<p:timing>` / `<p:bldLst>` resolves to a live `<p:cNvPr id>` (no animation orphans);
- **no `<p:timing>` contains an empty mainSeq `<p:childTnLst>`** (the silent repair trigger from §3.2);
- every `r:id`/`r:embed`/`r:dm`/`r:lo`/`r:qs`/`r:cs` in slide XML exists in that slide's `.rels`, and every `.rels` target file exists (no dangling refs);
- no duplicate `<p:cNvPr id>` within a slide's `spTree`;
- no `changesInfo` **or `revisionInfo`** part or reference remains (§3.1);
- **no `notesSlide` part is referenced by more than one slide** (§3.4b);
- **every `<a:t>` and `<dgm:t>` whose text has leading or trailing whitespace (including space-only and newline-only runs) carries `xml:space="preserve"`** (the silent repair trigger from §3.8);
- no orphaned notes slides (a `notesSlide` part referenced by no slide — remove the part, its rels and content-type override);
- every part is well-formed XML and `pack.py` validation passes.

**3.6 — QA caveat.** LibreOffice render QA does **not** surface §3.1/§3.2/§3.8 problems — it tolerates orphaned animations, ignores `changesInfo`, and renders whitespace-bearing runs fine whether or not `xml:space="preserve"` is present. Treat the §3.5 checks as mandatory programmatic gates *in addition to* visual QA. The only definitive open-test is PowerPoint itself.

**3.7 — Resize via `spPr/xfrm/ext`, never `.//ext`.** When moving or resizing any shape/picture, target the transform extent at `p:spPr/a:xfrm/a:ext` **explicitly**. A generic descendant search (`element.find('.//a:ext')`) returns the *first* `<a:ext>` in document order, which is frequently a blip/`extLst` extension (`<a:ext uri="…">`) rather than the transform — so the position updates but the size silently does not. Symptoms: a picture squeezed into its old frame (e.g. a landscape image left in a portrait box, which then overflows the slide and renders a blank thumbnail), or boxes that move closer together but keep their old height and overlap. Always resolve `xfrm = sp.find('p:spPr/a:xfrm')` then set on its direct `a:off`/`a:ext` children.

**3.8 — Preserve significant whitespace on text runs (the silent repair trigger).** A `<a:t>` or `<dgm:t>` (SmartArt cached text) whose content has **leading or trailing whitespace** — or which is **only** whitespace (a `" "` spacer run, a `"\n        "` newline run) — **must** carry `xml:space="preserve"`. Without it, PowerPoint sees a mismatch between the run as written and the run it would itself emit (it always preserves such runs) and offers to **repair** the deck on open. The repair is non-destructive — content renders correctly either way — but the prompt looks alarming and must not ship.

Why it's dangerous: it is **completely silent** to every check short of PowerPoint itself. LibreOffice renders identically; `pack.py` *auto-repairs* it on pack (it will report e.g. `Repaired: slideN.xml: Added xml:space='preserve' to a:t: 'Point 1: '`) — which means **splitting/re-packing a deck masks the bug**, because each re-pack silently fixes it. A deck can therefore open clean when split into halves yet prompt for repair as a whole, if the delivered whole was packed without the repair actually being written through.

Where it comes from, in order of frequency:
- **Runs Claude authors when pouring/adapting.** Any label or connective fragment with an edge space — `"Point 1: "`, `"Develop: "`, `"Video link: "`, `"Date: "`, `"Window partner: "`, leading-space continuations like `" and blocks of land may "`, and **blank spacer paragraphs** written as `<a:t> </a:t>`. Write these with `xml:space="preserve"` **at creation**, e.g. `<a:t xml:space="preserve">Point 1: </a:t>` — do not rely on `pack.py` to retrofit it.
- **Template runs inherited on transplant** — the house-style layouts (the Lesson-1 expectations list, the I/We/You-do label bands) and the **SmartArt `data1.xml` / `drawing1.xml`** newline runs carry the same latent issue; they ride along into the output.
- **Grafted live-shape runs** (§2.13) — when transplanting live shapes from a source slide, sweep them too.

Fix + gate: before packing, sweep **every** XML part (slides, layouts, masters, notes, **and `diagrams/*.xml`**) and add `xml:space="preserve"` to any `<a:t>`/`<dgm:t>` whose text `!= text.strip()` (this includes the whitespace-only runs — don't skip them, which was the miss that left `<a:t> </a:t>` and the `dgm:t` newlines behind on the first pass). Then assert zero remain (§3.5). **Verify on the *delivered* artifact**, not a fresh re-pack: read the output `.pptx` back and confirm no offending run survives, since a re-pack would hide a delivery that didn't actually persist the fix. *(Note: when verifying programmatically, scan **all** matches in a part, not just the first — a single-match check can read an unrelated run and miss the offending one.)*

**3.9 — Known template-quirk: strip the HS7 decorative arrow.** The **HS7** (info) template contains a decorative red right-arrow shape (`name="Arrow: Right 9"`) sitting between its two picture slots. When pouring HS7 content, **delete this arrow** unless the source genuinely has an arrow there — otherwise it floats over poured text/images (it surfaced over two-column text and over a 4-image grid on the NH5 deck). Apply to every HS7-based output slide.

## 4. Pipeline placement

Run order per source slide: **detect type (§1) → assign match confidence (§1.1)**. Then per deck: **present the full mapping table + check in on every ambiguous match (§1.1) → wait for user sign-off**. Only then, per slide: **transplant template (with all diagram/HD-photo parts §3.4; fix notes refs §3.4b) → pour content verbatim (writing whitespace-bearing runs with `xml:space="preserve"` §3.8; live-shape grafts per §2.13) → adapt body where the fit isn't clean (§0, flag `⚠ ADAPTED`) → apply content rules (§2: title font/placeholder §2.14, TPS icon §2.15, image align/maximise/verify §2.4, bold keywords §2.16, EQ-bar `fontScale` §2.11, table/MCQ handling §2.17, strip HS7 arrow §3.9) → structural cleanup (§3.1–3.4b) → fit-to-box (§2.5)**. Finally per deck: **whitespace-preservation sweep over all XML parts (§3.8) → validation gate (§3.5) → visual QA → deliver (verifying the gate on the delivered artifact, not a re-pack) → run the continuous-improvement loop (§5): write every user steer to memory and emit the generalized rule-bullets for the next version bump**.

This rule-set is the conversion authority for the master spec; the Activity Register and palette/layout sections of the spec are unaffected by it.

## 5. Continuous improvement loop *(every conversion)*

The rule-set improves only if every steer is captured. On **every** conversion that yields new routing or handling information from the user, run this loop (per §0 "always be learning"):

1. **During the build** — whenever the user answers a check-in, overrides a call, corrects a mapping mid-build, names a new icon→template signal, or states a handling preference, treat it as a candidate durable rule. Apply it to the current build *and* note it.
2. **Write to memory immediately** — record each steer as a concise memory (e.g. "Lesson converter — mini-whiteboard icon routes to HS20, not HS13") so the very next conversation benefits, before any formal rules release.
3. **Capture generalized bullets at the end of the conversation** — produce a short `Lesson converter —` prefixed bullet list (detection-signal → target template, or the handling policy), **topic- and slide-number agnostic**, each stating what it maps TO and what it should NOT be confused with. One bullet per decision; merge duplicates.
4. **Flag promotions vs refinements** — for each bullet, note whether it belongs in the **§1 taxonomy table** (a new detection-signal → template routing) or as a **§2/§3 policy refinement**.
5. **Flag contradictions for user resolution — never auto-resolve.** If a new steer conflicts with an existing rule, or if **memory and this rules document disagree**, raise it as `⚠ AMBIGUITY` and **have the user resolve it**. Do **not** silently overwrite either source, and do **not** assume one wins by default (neither "latest memory wins" nor "the doc wins"). Surface both versions, state the conflict plainly, and wait for the user's call before applying either (e.g. the HS9 "TPS header bars, not reveal" signal superseding an earlier "HS9 is only for reveal" reading was a user-confirmed reconciliation, not an automatic one).

**Memory ↔ document precedence.** Memory captures the most recent steers and the document is the deliberate, reconciled version; the two can drift. When they agree, either may be cited. When they **disagree**, it is an `⚠ AMBIGUITY` for the user (per step 5) — the converter must not pick a winner on its own. Memories are ordered by recency (later entries added more recently) but carry no explicit timestamps, so relative position is a hint, not a licence to override the document automatically.

These bullets are the input to the next version bump (vN → vN+1). Skip only pure content choices that imply no reusable policy (per the extraction guidance); everything that touches routing or handling gets captured.
